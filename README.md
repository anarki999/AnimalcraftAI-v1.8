## Update Log（更新日志）

### V10（July 23,2026）2026年7月23日

**搜索效率类**

**1. 空步裁剪（Null Move Pruning）**

让引擎在非危急局面下，模拟"让对方连走两步"，若对方仍无法翻盘，说明当前局面优势明显，可提早剪掉这条分支，加速搜索。

**棋力功用**：大幅减少无效分支的计算量，让引擎在同样时间内能把节省下来的算力投入更深层的搜索，等效提升实战中的思考深度。


**2. 前置裁剪（Futility Pruning）**

在剩余深度仅1~3层、非危急局面时，若静态评估加上该深度的最大可能改善幅度仍追不上alpha，直接放弃深入搜索该节点，节省大量无意义运算。

**棋力功用**：专门加速搜索树末端（叶节点附近）的处理效率，让引擎能把省下的时间分配到更关键的中局分支，间接提升整体搜索深度与速度。


**3. 浮动比例削减后期走法裁剪（Late Move Reduction，v7升级版）**

原本固定砍1层深度，改为用 log(深度)×log(着法序号) 算出浮动削减量；历史表分数高的走法（过去表现佳）削减力度会减轻，让排序靠后的"冷门"走法搜得浅、热门走法搜得深。

**棋力功用**：比固定砍1层更精准地分配算力——不会因排序机制的偶然误差而错杀潜在好棋，同时对明显的差棋加大削减幅度，让引擎在有限时间内把资源集中在真正有价值的分支上，兼顾速度与正确性。


**4. 价值裁剪（Delta Pruning，用于静态搜索）**

在Quiescence吃子搜索中，若某个吃子走法即使成功也不可能把分数拉回alpha以上，直接跳过该分支，减少无意义的深搜。

**棋力功用**：让静态搜索（处理吃子连锁反应的关键环节）更快收敛，避免在明显无望的吃子线上浪费算力，确保引擎能在时间限制内把战术计算做得更完整。


**5. 渐进式重搜望孔窗口（Aspiration Window，v7升级版）**

迭代加深时不再一次退回全窗口重搜，而是以每次±200分的幅度逐步放宽视窗，多数情况1~2次微调即可收敛，减少fail-low/fail-high导致的重复计算。

**棋力功用**：本身不改变引擎的判断准确度，而是用更省时的方式达到同样正确的结果——省下的时间可转化为多算1层甚至半层的深度，在固定时限的对局中，搜索深度每增加1层通常能带来明显的棋力提升；同时避免大幅重搜造成的耗时波动，让迭代加深节奏更稳定，也更利于Lazy SMP多线程下把节省的算力有效转化为整体吞吐量提升。


**6. 将军延伸（Check Extension）**

若某一步直接威胁入穴（叫杀），搜索深度不减1层，让引擎多算一层确认这步棋的真实性，避免误判。

**棋力功用**：防止引擎在搜索深度受限时，把"看似能叫杀但实际上是虚招"的走法误判为好棋，提升关键战术局面下的判断可靠度，减少因搜索深度不足导致的漏算。


**7. 后台思考模式（Background Thinking / Infinite Mode）**

新增 go infinite 指令支持，透过 bInfinite 旗标让引擎在没有时间上限的情况下持续迭代加深搜索，直到收到GUI发出的 stop 指令才停止；期间引擎会持续透过 info depth、info time nodes 等讯息即时回报当前搜索进度与节点数，供GUI端同步显示。

**棋力功用**：让引擎能在对手思考、或轮到玩家走棋的空档持续分析局面（也就是象棋引擎常说的"后台思考"/Ponder），不浪费任何可用的运算时间；配合UCCI协议的 stop 指令，GUI可以随时打断分析并立即取得当下最佳结果，兼顾深度搜索与响应速度，让引擎在实际对局中能更充分利用每一秒可用时间，变相提升有效搜索深度与棋力表现。

**8. 单一延伸与反向多切裁剪（Singular Extension & Reverse Multi-Cut，v10新增）**

针对零窗口测试(PVS)结果"差一点点被剪掉"的走法（分数落在alpha下方SINGULAR_MARGIN=30分的窄区间内），且该走法排序在前REVERSE_MULTICUT_RANK_LIMIT=3名以内时，用多算SINGULAR_EXT_DEPTH=2层的深度重新做一次零窗口验证，通过才升级为完整窗口确认精确分数；其余排序靠后的走法维持原本裁剪效率不受影响，只对"有嫌疑"的候选花额外算力。

**棋力功用**：专门解决零窗口测试对"弃子换攻杀"、"多子vs多兵"这类延迟收益招法容易误判fail-low的问题，避免真正的好棋因浅层评估分数暂时偏低而被过早剪掉；由于只对排序靠前、少数可疑的走法做二次确认，额外开销可控，能在不明显拖慢整体搜索速度的前提下，提升引擎对战术性、延迟收益招法的判断准确度。

**评估精准度类**

**9. 威胁度评估（Threat Evaluation）**

新增 EvalThreat() 函式，逐一检查己方每颗棋子是否正被对方棋子合法威胁（含跳河攻击），依棋子等级给予不同扣分幅度（象扣最重、猫扣最轻），让静态评估更贴近真实战术局面。

**棋力功用**：让引擎在尚未深入搜索前，就能对"棋子正被威胁"的局面提前警觉并降低评分，减少因单纯比较子力价值而忽略战术风险所导致的误判，尤其能提升引擎在防守与应对威胁时的判断品质。


**10. 简化版修正历史机制（Correction History Lite，v8新增）**

用一个仅8格的小数组，按棋子种类记录"本层与上层分数落差"的指数滑动平均(EMA)修正值，微调Evaluate()的静态评估，帮助引擎更快收敛到准确评价，且几乎零额外开销。

**棋力功用**：让静态评估随着搜索层数推进逐步自我校正，减少评估函式的系统性偏差，使引擎对局面优劣的判断更快趋于准确，尤其有助于改善浅层搜索时的评分稳定性，且几乎不增加额外运算负担。

**11. 兽穴威胁与进攻潜力评估（Attack Potential Evaluation，v10新增）**

新增 EvalAttackPotential() 函式，针对象、狮、虎（进攻力最强的三种棋子），依其与敌方兽穴的曼哈顿距离给予额外加分：距离≤1格（下一步就能进穴，近似绝杀威胁）加70分，距离≤3格（逼近兽穴的威压姿态）加25分，并整合进Evaluate()的最终评分。

**棋力功用**：改善静态评估对战术潜力的敏感度，避免引擎把"弃子换攻势"或"逼近敌方兽穴"这类招法，单纯因为当下子力减少而评得太低；让评估函式本身就更贴近真实的攻杀价值，从源头减少零窗口测试因评分偏低而误判剪掉正解的机率，与Singular Extension形成互补。


**多线程架构类**

**12. 惰性对称多处理多线程搜索（Lazy SMP）**

将 mvResult、历史表、杀手走法表等改为thread_local独立变量，各线程各自搜索、共享同一份置换表(HashTable)，透过atomic变量同步停止讯号与节点计数，充分利用多核CPU加速搜索。

**棋力功用**：让引擎能真正发挥多核CPU的硬体性能，多个线程各自从不同角度探索局面并共享置换表成果，等于用更多算力在同样时间内覆盖更广、更深的搜索范围，是棋力提升最直接且最具扩展性的一项优化。


**记忆体优化类**

**13. 置换表容量调整（Hash Table Size Adjustment）**

HASH_SIZE 从 1<<25 缩减为 1<<20（约100万项），避免记忆体占用过大。

**棋力功用**：属于资源管理优化，本身不直接提升单次搜索的判断力，但能避免记忆体超额占用拖累系统效能（尤其在多线程同时运行、记忆体需求叠加的情境下），确保引擎在实际运行环境中维持稳定速度。


**14. 置换表项结构体字段精简（HashItem Struct Field Compaction）**

评估分数字段由int改为short，缩小每个置换表项的记忆体大小，让同样记忆体容量能存放更多项目、减少置换表冲突。

**棋力功用**：在同样记忆体预算下容纳更多置换表项目，降低不同局面的杂凑碰撞（Hash Collision）机率，让置换表的"记忆效果"更可靠——引擎能更准确地重复利用之前算过的局面结果，间接提升搜索效率与深度。


**架構與穩定性修复类**

**15. 异步搜索线程棋局同步修复（Async Search Thread Position Sync Fix）**

之前将 SearchMain 改为丢到独立线程 g_searchThread 执行，让主程式能持续读取 stop/ponderhit 指令，但棋局资料 pos 在每条线程里都是各自独立的一份，新线程启动时并未从主线程把当前真实棋局复制过去，导致该线程拿到的其实是一个从未摆过任何棋子的空棋盘，在这个空棋盘上搜索自然找不到任何有效走法。修复方式为在派发新线程之前，先让主线程把当下棋局的完整快照传给 SearchMain，确保搜索线程一启动就基于正确的棋局进行计算。

**棋力功用**：属于关键正确性修复，解决了引擎在异步化搜索架构下"算出无效着法、bestmove 输出乱码"的严重缺陷，确保每次 go 指令启动的搜索都基于真实棋局进行，是多线程UCCI通信改造能够正常运作的前提。

**16. Stop指令即时响应修复（Immediate Stop Response Fix）**

SearchFull 原本在判断是否要提前中断搜索时，误加了一个"只有非主搜索线程才生效"的额外条件，导致负责执行主要搜索流程的那条线程完全不会检查停止信号，收到 stop 指令后仍会把当前这一整层的搜索跑完（深度较高时可能耗时数秒至数十秒）才真正停下。修复方式为移除这个多余的条件，让所有线程一律平等地检查停止信号，并在根节点走法枚举的迴圈（SearchRootWindow）里额外加入检查点，让停止指令能在每算完一个候选走法后就立刻生效。

**棋力功用**：让引擎在实战与GUI操作时能真正做到"随时可打断"，避免因深度暴增、节点数暴涨（如深度12单层即耗费近1000万节点）而卡住数秒才响应停止指令，大幅改善引擎的实用性与操作体验，避免因响应延迟被判超时或误判引擎当机。

**17. Ponderhit指令即时响应修复（Immediate Ponderhit Response Fix）**

ponderhit 指令原本只重设计时起点并解除"后台思考不计时"的豁免状态，但引擎检查时间是否用完的动作，原本只安排在每一层搜索完整跑完之后才执行一次。如果 ponderhit 送达时刚好碰上正在进行一层特别深、特别耗时的搜索（后台思考期间因不受时间限制可以无限深搜），就会导致时间早已用完，却要等这一层彻底跑完才真正停下来。修复方式为新增一个独立的超时检查，并同样加进 SearchRootWindow 的根节点走法迴圈里，让主线程在每算完一个候选走法时都主动确认时间是否已经用完，一旦超时就立刻发出全域停止信号，让搜索马上结束。

**棋力功用**：让"后台思考(Pondering)→对手真实落子→ponderhit"这一实战最常见的流程，能确保引擎在剩余时间预算耗尽时准确、及时地给出 bestmove，而不会因pondering期间累积的超深搜索状态而拖长响应时间，是让第7项"后台思考模式"真正能安全落地于实战对局的必要配套修复。
